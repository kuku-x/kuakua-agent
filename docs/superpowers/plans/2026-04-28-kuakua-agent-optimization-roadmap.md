## 当前产品判断

夸夸 App 已经具备一条基本可用链路：

- 电脑使用行为记录
- 手机使用行为同步
- 每日汇总与应用分布展示
- 晚间总结生成
- 主动夸夸机制
- 聊天陪伴入口
- 通知与语音播报基础链路

现阶段最重要的，不是继续横向铺功能，而是把产品收拢为一个更稳定、更持续、更有陪伴感的工具：

- 看得见每天状态变化
- 能给出合适的鼓励
- 能帮助回顾和复盘
- 在低谷时不过度打扰

**一句话定位**：夸夸 App 不是"又一个效率工具"，而是"以行为数据为基础的长期陪伴型桌面应用"。

## 产品重心决策

### 桌面端是主产品端

后续主要体验统一放在桌面端承载。

原因：

- 晚间总结、历史回看、时间线、聊天陪伴都更适合桌面端。
- 桌面端更适合承载完整信息和沉淀式回顾。
- 在当前资源下，做深一个端比同时打磨双端更划算。

### 手机端保留，但降级为数据侧能力

手机端继续保留，但角色明确为：

- `手机使用数据采集与同步工具`

而不是：

- `需要持续扩展体验的独立产品端`

这意味着：

- 不删除手机端工程
- 不下线手机同步能力
- 不主动新增手机端产品功能
- 手机端只做必要修复、兼容和同步稳定性维护

### 端侧分工结论

- `手机端负责提供数据`
- `桌面端负责提供核心体验`

## 维护边界

### 手机端继续做的事

- 同步失败修复
- 权限问题修复
- 必要兼容性修复
- 与桌面端统计口径明显不一致时的关键修复

### 手机端暂不做的事

- 新页面扩展
- 独立复杂可视化
- 深度陪伴体验打磨
- 与桌面端逐项功能对齐

### 任何需求进入开发前都先问一句

这个需求是在增强"桌面端核心体验"，还是在继续摊薄到双端？

如果是后者，默认不做，除非它直接提升数据质量或稳定性。

## 优化原则

后续所有需求按以下原则排序：

1. 优先增强真实陪伴感，而不是堆功能。
2. 优先增强"每天都能用到"的能力，而不是低频展示能力。
3. 优先补全"记录 -> 理解 -> 鼓励 -> 建议 -> 回看"的闭环。
4. 优先低摩擦体验，尽量少配置、少打断、少学习成本。
5. 优先提升数据可信度、通知可靠性、语音可诊断性。
6. 优先做桌面端最直接有感的改进。

## 路线图结构

为避免"想做的很多、真正落地的很少"，后续需求分为三层：

- `P0 基础体验`：不做会影响产品成立感或日常使用。
- `P1 增强体验`：能明显提升留存和陪伴感，但依赖 P0 稳定。
- `P2 特色探索`：可能成为亮点，但不应抢占主链路资源。

只有满足前一层基本稳定，才进入下一层。

### 1. 每周复盘

建议系统自动生成每周总结，包括：

- 本周最专注的一天
- 最容易分心的时段
- 手机娱乐变化趋势
- 本周最值得被夸的一件事
- 下周建议

**价值**：日总结解决"今天"，周复盘解决"趋势"

### 2. 更智能的主动夸夸

建议从单点规则触发，逐步升级为结合上下文的鼓励：

- 连续专注时长
- 最近几天变化趋势
- 手机和电脑切换节奏
- 晚间状态判断：夸、提醒、还是安慰

**价值**：更像在观察用户，而不是机械触发

### 3. 应用分类自定义

建议支持：

- 手动定义工作 / 娱乐 / 其他
- 包名映射
- 进程名映射

**价值**：提高统计准确度，让建议更贴近个人习惯

### 4. 手动补记

建议支持记录自动采集不到的重要行为：

- 运动
- 读书
- 见面
- 散步
- 开会

**价值**：避免"没被记录就像没发生"，让总结更完整

### 5. 全局错误可诊断性

建议统一错误提示入口：

- TTS 播报失败时在首页或设置页显示状态指示灯
- 区分错误类型：Fish Audio Key 无效、网络不通、Kokoro 未安装、播放失败
- 定时健康检查并在异常时温和提醒

**价值**：让用户感知到问题而非静默失败，降低排错成本

### 6. 设置页 TTS 试听按钮

建议在设置页语音配置区域增加一键试听：

- 选择音色后可直接试听固定短句
- 试听失败时在页内显示错误原因

**价值**：完成"测试语音播报"的最后一环，让语音配置可感知、可验证

### 7. 应用分类记忆

建议支持用户手动标记应用分类并持久化：

- 右键或点击将某个 App 标记为工作 / 娱乐 / 其他
- 分类结果存入本地，覆盖内置关键词匹配
- 新安装应用首次出现时提示分类

**价值**：解决 Claude 等新 App 需要手动加关键词的问题，让分类越用越准

### 8. 异常日提醒

建议在出现明显偏差时给出温和提醒：

- 手机娱乐暴涨
- 长时间没休息
- 凌晨仍持续活跃
- 明显比平时更分散

**价值**：从"夸夸工具"升级为"节奏守护工具"

## P2 特色探索

这些不是当前主链路，但可能形成产品个性。

### 1. 情绪化陪伴模式

- 温柔型
- 元气型
- 冷静陪伴型
- 严格但不凶型

### 2. 轻量目标系统

- 今天专注 2 小时
- 11 点前放下手机
- 本周少刷短视频

### 3. 成就系统

- 连续几天有晚间总结
- 连续几天出现专注段
- 娱乐使用趋势下降

### 4. 低谷保护模式

- 降低命令式建议
- 提高陪伴性语言
- 更重视恢复而不是推动

### 5. 夸夸回忆录

- 回看过去状态好的某一天
- 提醒"你以前做到过"

## 持续性专项

这些不是单一功能，而是应持续穿插推进的底层体验工作。

### 1. 通知可靠性

- 通知来源统一
- 点击后能打开对应页面
- 通知与应用内卡片协同

### 2. 配置可诊断性

- 同步、语音、模型、通知都有状态检测
- 错误定位尽量直接给到用户

### 3. 数据可信度提示

- 明确今天的数据来自电脑、手机或混合
- 某一端缺失时要提示

### 4. 初始化体验

- 首次启动引导尽量清晰
- 帮助完成 ActivityWatch、手机同步、语音配置

### 5. TTS 语音播报 ✅ 已完成基础链路

- Fish Audio（云端高质量）作为主引擎，Kokoro-82M（本地）作为备用
- `FallbackTTS` 自动切换：Fish Audio → 失败回退 Kokoro
- 设置页支持引擎选择和音色切换
- 晚间总结和主动夸夸两条主链路均已接入 TTS
- 具体见 [本地 TTS 迁移计划](./2026-04-28-local-tts-migration-plan.md)

## 当前建议实施顺序

P0 已全部完成。P1 剩余 6 项，建议按以下顺序推进：

1. 应用分类记忆 — 直接解决 Claude 等新 App 被误分类的问题
2. 每周复盘 — 晚间总结已有沉淀，周复盘条件成熟
3. 更智能的主动夸夸 — 结合趋势数据的上下文鼓励
4. 异常日提醒 — 从夸夸升级为节奏守护
5. 应用分类自定义 — 手动标记覆盖内置规则
6. 手动补记 — 补充无法采集的行为

**前提**：`每周复盘` 不应早于了晚间总结和历史归档（✅已满足）。

## 需求进入开发的检查清单

以后新增需求时，先过这 6 个问题：

1. 这个需求是在增强桌面端核心体验吗？
2. 它是否直接提升"记录、理解、鼓励、建议、回看"闭环？
3. 它是否会明显提升日常可感知价值？
4. 它会不会扩大手机端维护负担？
5. 它是否依赖尚未稳定的前置能力？
6. 做完之后，用户能明显感知到差异吗？

如果前四项多数为"否"，默认不进入近期开发。

## README 与对外表达建议

后续 README、设置页、设计文档中，建议逐步统一以下表述：

- 手机端是辅助同步工具
- 桌面端是主使用入口
- 产品核心价值是"陪伴与回顾"，不是"参数和统计项堆叠"

这样可以减少预期错位，也能降低未来再次回到"双端都想做完整"的维护风险。

## 结论

P0 基础体验全部落地。当前焦点转入 P1：

- **应用分类记忆** — 让数据更准
- **每周复盘** — 从日到周的陪伴升级
- **更智能的主动夸夸** — 从机械触发到上下文感知

后续只要持续围绕主链路推进，夸夸 App 才会越来越像一个真正长期可用的陪伴型产品。
