# 夸夸 App 后续优化与维护路线图

## 文档目的

这份文档用于统一记录夸夸 App 的后续优化方向，以及当前已经明确的产品维护策略。

后面继续迭代时，尽量只维护这一份路线图，避免产品方向、功能优先级、端侧定位散落在多份文档里。

## 当前产品判断

夸夸 App 现在已经有了不错的基础：

- 电脑使用行为记录
- 手机使用行为同步
- 每日汇总与应用分布展示
- 晚间总结生成
- 主动夸夸机制
- 聊天陪伴入口
- 通知与语音播报基础链路

下一阶段最重要的，不是继续横向铺很多功能，而是把产品重心收拢，让它更像一个：

- 看得见你每天状态变化
- 愿意鼓励你
- 能帮你回顾和复盘
- 在低谷时不过度打扰

的长期陪伴型应用。

## 产品重心决策

### 桌面端是主产品端

后续产品体验的主要承载端，明确放在桌面端。

主要原因：

- 晚间总结、历史回看、语音播报、聊天陪伴都更适合在桌面端完成。
- 桌面端更适合承载完整信息展示和沉淀式回顾。
- 继续把桌面端做深，比双端同时打磨更划算。

### 手机端保留，但不再作为重点产品端

当前阶段，手机端保留，但定位调整为：

- `手机使用数据采集与同步工具`

而不是：

- `需要持续打磨和演进的独立产品端`

这意味着：

- 暂不删除手机端工程
- 暂不下线手机同步能力
- 不再主动投入手机端新功能开发
- 手机端只做必要修复和最低限度维护

### 手机端后续维护边界

会继续做的：

- 同步失败修复
- 权限和配置问题修复
- 必要兼容性修复
- 与桌面端统计口径明显不一致时的关键修复

暂不再做的：

- 手机端新页面扩展
- 手机端独立复杂可视化
- 手机端陪伴体验深度打磨
- 手机端和桌面端功能对齐式开发

### 端侧分工结论

后续建议明确成一句话：

- `手机端负责提供数据`
- `桌面端负责提供主要体验`

## 优化原则

后续所有优化，建议遵守以下原则：

1. 优先增强真实陪伴感，而不是堆参数和功能。
2. 优先增强“每天都能用到”的能力，而不是偶尔打开一次的能力。
3. 优先增强“记录 -> 理解 -> 鼓励 -> 建议 -> 回看”的闭环。
4. 优先做低摩擦体验，尽量少配置、少打断、少学习成本。
5. 优先做桌面端价值最直接的改进，避免精力被双端分散。

## 近期优先项

这些是最值得优先做的一批。

### 1. 完整晚间总结页

当前晚间总结已经能生成、通知、在应用内展示卡片，但还不够沉淀。

建议补充：

- 独立的晚间总结详情页
- 可查看完整内容，而不是只靠顶部卡片
- 支持回看历史晚间总结

价值：

- 用户可以真正“保存今天”
- 晚间总结从提醒升级为长期陪伴记录

### 2. 晚间总结历史归档

晚间总结不应该只活在当天通知里。

建议补充：

- 历史列表
- 按日期回看
- 可筛选“夸夸 / 建议 / 状态波动”

价值：

- 让夸夸 App 更像成长记录工具

### 3. 每日时间线

当前已有总时长和应用分布，但缺少“今天是怎么走过来的”。

建议补充：

- 按时间展示今天的主要行为片段
- 展示专注、切换、休息、手机高频使用时段
- 标记明显的节奏变化

价值：

- 用户能更具体地理解“今天发生了什么”
- 更适合生成有针对性的夸夸和建议

### 4. 测试语音播报按钮

当前语音配置是否可用，需要靠真实触发来验证，排错成本高。

建议补充：

- 设置页新增“测试语音播报”
- 直接播一段固定短句
- 如果失败，明确提示是模型路径、voice、本地 TTS 初始化还是播放问题

价值：

- 大幅降低语音相关排错成本
- 提升功能可感知性

### 5. 手机和电脑统一视图继续打磨

虽然手机端不再重点维护，但手机数据对桌面端仍然重要。

建议补充：

- 统一应用名显示规范
- 统一 top apps 展示逻辑
- 支持按设备切换查看
- 支持查看“电脑 / 手机 / 全部”三个维度

价值：

- 降低理解成本
- 提升数据可信度和一致性

## 中期重点方向

这些更适合在当前基础稳定后推进。

### 1. 每周复盘

建议系统自动生成每周总结：

- 本周最专注的一天
- 本周最容易分心的时段
- 手机娱乐变化趋势
- 本周值得被夸的一件事
- 下周建议

价值：

- 日总结解决“今天”，周复盘解决“趋势”

### 2. 主动夸夸更智能

当前主动夸夸主要还是规则驱动。

建议后续增强：

- 结合连续专注时长
- 结合最近几天趋势
- 结合手机和电脑的切换节奏
- 结合晚间状态判断该夸、该提醒、还是该安慰

价值：

- 更像真的在观察你，而不是定时触发一句话

### 3. 应用分类自定义

系统默认分类不一定适合每个人。

建议补充：

- 用户可手动定义应用属于工作 / 娱乐 / 其他
- 支持包名映射和进程名映射

价值：

- 提高统计准确度
- 让建议更贴近个人习惯

### 4. 手动补记

有些重要行为不会被电脑或手机自动记录。

建议补充：

- 手动记录运动、读书、见面、散步、开会等
- 这些内容参与晚间总结

价值：

- 避免“没被电脑记录就像没发生”
- 让总结更完整、更有人味

### 5. 异常日提醒

建议在检测到明显偏差时，主动给出温和提醒：

- 手机娱乐时间暴涨
- 长时间没有休息
- 凌晨还在持续活跃
- 今天明显比平时更分散

价值：

- 从“夸夸工具”升级为“节奏守护工具”

## 远期产品亮点

这些不是当前最急，但很可能成为产品特色。

### 1. 情绪化陪伴模式

不同的人喜欢不同的鼓励风格。

可以考虑支持：

- 温柔型
- 元气型
- 冷静陪伴型
- 严格但不凶型

价值：

- 让陪伴更个性化

### 2. 目标系统

建议支持用户自己设定轻量目标：

- 今天专注 2 小时
- 晚上 11 点前放下手机
- 本周少刷短视频

价值：

- 把夸夸从“记录结果”延伸到“陪你完成目标”

### 3. 成就系统

建议记录可持续的小成就：

- 连续 3 天有晚间总结
- 连续 5 天早上有专注段
- 手机娱乐下降趋势达成

价值：

- 增强持续使用动力

### 4. 低谷保护模式

不是所有时候都适合“元气鼓励”。

建议在识别到低状态时：

- 降低命令式建议
- 提高陪伴性语言
- 更重视放松和恢复

价值：

- 避免机械化鼓励带来的反效果

### 5. 夸夸回忆录

建议周期性回顾过去表现好的某一天：

- 你之前也做到过
- 你最好的状态是什么样
- 你那天是怎么开始的

价值：

- 在低谷时更有恢复力

## 可用性与稳定性优化

除了新功能，稳定性和体验也值得持续打磨。

### 1. 通知可靠性

- 系统通知来源统一
- 点击通知后明确打开对应页面
- 通知和应用内展示协同工作

### 2. 配置可诊断性

- 所有关键能力都提供状态检测
- 同步、语音、模型、通知都能快速定位失败原因

### 3. 数据可信度提示

- 告诉用户今天数据来自哪里
- 标注电脑、手机、还是混合
- 当某一端缺失时给出提醒

### 4. 初始化体验

- 新用户首次启动时有清晰引导
- 引导完成 ActivityWatch、手机同步、语音配置

### 5. 本地语音替代云端

- 逐步减少对付费 TTS 的依赖
- 优先把晚间总结和主动夸夸迁移到本地 TTS
- 已选定 Kokoro-82M（Apache 2.0，82M 参数，CPU 实时推理），见 [本地 TTS 迁移计划](./2026-04-28-local-tts-migration-plan.md)

## 建议优先级排序

如果只选一小批最值得尽快做的，建议顺序如下：

1. 完整晚间总结页
2. 晚间总结历史归档
3. 每日时间线
4. 测试语音播报按钮
5. 应用分类自定义
6. 每周复盘

## README 和文档表达建议

后续在 README、设置页、设计文档中，建议逐步统一这层表达：

- 手机端是辅助同步工具
- 桌面端是主使用入口

这样可以减少预期错位，也能避免未来再次陷入“双端都想做完整”的维护负担。

## 结论

夸夸 App 后续最有价值的方向，不是做成“又一个效率工具”，而是做成一个：

- 看得见你每天状态变化
- 愿意鼓励你
- 能帮你回顾和复盘
- 在低谷时也不打扰过度

的长期陪伴型应用。

当前最合理的产品策略是：

- 保留手机端
- 保留手机同步能力
- 停止手机端功能扩展
- 将产品重心明确切到桌面端

后续迭代建议优先围绕：

- 晚间总结沉淀
- 时间线理解
- 个性化陪伴
- 更稳定的语音和通知

逐步推进。
